### IJ-SEQEXPR — `(e1;e2;e3)` value-of-last with α/β semantics (`3f23d132`)

TT_SEQ_EXPR was lowering to IR_SEQ which always returns FAILDESCR —
correct for proc-body fall-off, wrong for `x := (1;2;3)` which should
bind x=3. Listed as a latent bug in the 2026-05-17 part 3 handoff.

Four coherent edits:

1. **`src/include/IR.h`** adds IR_SEQ_EXPR after IR_SWAP.
2. **`src/lower/ir_exec.c`** adds IR_SEQ_EXPR executor right after
   IR_SEQ. Implements α/β generator semantics:
     - α (state==0): evaluate e1..e_{n-1} for side effect (any FAIL ⇒
       whole seq fails); then drive e_n fresh, set state=1.
     - β (state==1): resume e_n only — head statements DO NOT re-fire.
       When e_n exhausts, state→0, ω.
   This matches the side-effect-once / pump-last semantics needed by
   `every (x := x+10; "a" | "b" | "c")` (rung16) — x increments once,
   the alt pumps three times.
3. **`src/lower/lower_icn.c`** TT_SEQ_EXPR lowering switched from
   IR_SEQ to IR_SEQ_EXPR. Single-statement blocks are unwrapped by
   the parser (parse_block_or_expr) so n>=2 in practice. Proc bodies
   are NOT built via TT_SEQ_EXPR (parser keeps them as flat children
   of TT_FNC, wrapped in IR_SEQ by lower_icn_proc_body) — fall-off-end
   semantics preserved.
4. **Loop drivers IR_EVERY, IR_WHILE, IR_UNTIL** now do
   `nd->c[1]->state = 0` before each body call. Without this reset,
   a body block carrying state=1 from a prior iteration would skip
   its head statements on iteration 2+ (rung35 block_body_every_do_
   block / block_body_while_do_block regression caught & fixed in
   the same edit). Parallels the existing `nd->c[0]->state=0` reset
   on the condition side of WHILE/UNTIL.

Plus IR_SEQ_EXPR entry in `kind_names`.

**Result:** ir-run 187 → 189 (+2). rung20 seqexpr_basic + seqexpr_side
PASS. rung16 seqexpr_gen_basic remains PASS (verified: regressed without
α/β; correct with). rung35 block_body_every_do_block + while_do_block
remain PASS (verified: regressed without body-state-reset; correct with).

### IJ-INITIAL — `initial expr` one-shot via ival3 flag (`1fb213db`)

TT_INITIAL was folded into the TT_GLOBAL/TT_LOCAL/TT_STATIC_DECL bucket
— all four emitted IR_SUCCEED → `initial expr` was silently dropped.
This blocked rung21_initial_once and rung25_initial_once which expected
the static-like 11/12/13 sequence across three calls to a procedure
with `local x; initial x := 10; x := x + 1; write(x); end`.

Four coherent edits:

1. **`src/include/IR.h`** adds IR_INITIAL after IR_SEQ_EXPR.
2. **`src/lower/ir_exec.c`** adds IR_INITIAL executor right after
   IR_SUCCEED. Uses `nd->ival3` as has-run flag — specifically chosen
   because ival3 is the one mutable per-node field that IR_reset does
   NOT clear AND IR_snapshot_state/IR_restore_state do NOT copy, so it
   survives both the per-call reset and recursive snapshot/restore
   around inner same-proc calls. First entry: run c[0], set ival3=1.
   Subsequent entries: no-op. Always succeeds via γ with NULVCL.
3. **`src/lower/lower_icn.c`** splits TT_INITIAL out of the scope-decl
   bucket. Lowers c[0] (the assignment or block expression) and wraps
   in IR_INITIAL with ival3=0. Degenerate (n=0) falls back to
   IR_SUCCEED. Common AST shapes:
     - `initial x := 10;` → TT_INITIAL(c[0]=TT_ASSIGN(x,10))
     - `initial { s1; s2; ... };` → TT_INITIAL(c[0]=TT_SEQ_EXPR(s1,...))
4. **Persistence across calls "just works"** thanks to existing
   `build_proc_scope` logic in `src/lower/lower.c` (lines 1497-1513):
   it already removes any var initialised by a TT_INITIAL+TT_ASSIGN
   combo from the local scope. With the var absent from FRAME.sc, both
   IR_VAR reads and IR_ASSIGN writes fall through to global NV_GET/
   NV_SET. Combined with IR_INITIAL firing exactly once, this yields
   x persisting across calls — call 1 writes 10 to NV then 11; call 2
   reads NV=11 writes 12; call 3 reads NV=12 writes 13. No further
   code changes needed.

Plus IR_INITIAL entry in `kind_names`.

**Result:** ir-run 189 → 191 (+2). rung21_initial_once and rung25_
initial_once PASS. All gates hold.

## Open Notes / Carry-Forward

1. **IJ-INITIAL relies on build_proc_scope's existing initial-var
   removal.** That logic is in `src/lower/lower.c` lines 1497-1513.
   It pattern-matches `TT_INITIAL/TT_ASSIGN/TT_VAR` shape exactly. If
   future work changes the AST shape of `initial` (e.g. wrapping its
   body in a TT_SEQ_EXPR even for the single-assign case), the scope
   removal logic must be updated in lockstep. A regression would
   manifest as rung21/rung25 reverting to printing 11,11,11.

2. **IJ-SEQEXPR's body-state-reset in loop drivers** is a
   generally-useful invariant: a statement-context body block should
   not retain generator state across iterations. It's possible (but
   not verified) that other constructs would benefit from the same
   pattern — e.g. `if-then-else` with an SEQ_EXPR `then`-branch
   resumed across multiple if-evaluations. Watch for it.

3. **Two scrip drivers wrote slightly conflicting fixes to swap-related
   code in the SM path (separate ticket).** The IR-path is now clean;
   the SM path's `sm_swap` handler still exists at runtime. If/when
   SM-run is brought back online for Icon, verify they don't disagree
   on Icon-specific corner cases (e.g. swap of a TT_FIELD lvalue,
   which neither path supports today and which currently FAILs cleanly).

4. **IR_INITIAL's ival3 flag survives recursive entry.** I tested this
   was the design intent. If a future change adds ival3 to the
   IR_node_state_t snapshot struct (`src/include/IR.h:158`), the flag
   would get restored to its caller-time value mid-snapshot-pop, which
   could re-fire the initial. Don't add ival3 to that struct unless
   you also add a special-case skip for IR_INITIAL.

5. **rung16 seqexpr_gen_basic** was passing pre-change by coincidence
   (old IR_SEQ returned FAIL which short-circuited every). Now passes
   for the right reason (α/β semantics). Listed here in case anyone
   bisects through this session looking for the seqexpr-gen win.
