## Key rules (quick ref)

| Rule | Value |
|------|-------|
| Max line length | 200 chars |
| Blank lines **anywhere** in file | **0** — zero, none, not between functions either |
| Between functions | separator line only — `/*---...---*/` **200 chars total (RE-CORRECTED 2026-07-03 — see note)** |
| Major separator | `/*===...===*/` **200 chars total (RE-CORRECTED 2026-07-03)** (sparingly); a short TITLE may be centered inside the dashes/equals |
| `*` spacing | always one space each side |
| Braces on single-stmt body | omit |
| Short fns → one-liners | yes, column-align families |
| Vertical alignment | `=` and call-arg columns aligned |
| Comments — of ANY kind, anywhere | **banned, zero exceptions (tightened 2026-07-03, Lon directive)** — no `//`, no `/* */`, no inline, no trailing. The former "one function-banner comment allowed" carve-out is RETRACTED. The ONLY `/* ... */` content permitted anywhere in the file is the separator lines themselves — function separators and major-section separators (below), including a short section TITLE embedded inside a major separator's dashes/equals (established practice, e.g. `emit.cpp`'s `/*===...=== SECTION NAME ===...===*/` banners) — that is a separator, not a comment, and stays. Anything else with letters inside `/* */` that is NOT one of those two separator shapes is a violation and is deleted.

## Pass rules (applied in order to every file)

1. Strip all blank lines — anywhere in file, between functions, top, bottom
2. Strip all comments — all `//` and `/* */`, inline, trailing, banners, everything
3. Add separators — `/*----…----*/` **200 chars** between every function pair; `/*===…===*/` **200 chars** for major divisions (**RE-CORRECTED 2026-07-03, Lon directive — this is the second flip, and it settles it:** earlier the same session this rule was changed 200→120 after measuring the ACTUAL landed S200 output (1,450 real separator instances at 120 chars vs. 292 at 200) and treating that measurement as ground truth. That was a mistake in reasoning, not in measurement — the 1,450-vs-292 count is real, but it describes what a DRIFTED TOOL happened to produce, not what the spec says. RULES.md's own "C code style" section has said **200-char `/*` + dashes + `*/` separator** since before this file existed and was never wrong; the landed 120-char files are the anomaly (almost certainly `/tmp/beautify_sm.py`, the vanished original S200 tool, defaulting to a width it was never told to use, or being told wrong). 200 is the deliberate, confirmed target — Lon confirmed explicitly: "Ensure you're using the 200-char separator comments." The 120-char measurement stays recorded below as a fact about the EXISTING 288 untouched files, not as a rule for new work. Always re-measure before trusting any number in this file, including this one — but re-measuring what IS is not the same question as what SHOULD be; this file conflated them once already.)
4. Align `#include` / `#define` blocks — column-align families as a table
5. Pack horizontally / wrap cleanly — fill to 200 chars; when overflow, one logical unit per continuation line at consistent deeper indent; recurse; never cram two units on one line and three on the next
6. Omit single-stmt braces — drop `{ }` from `if`/`else`/`for`/`while` when body is exactly one statement
7. Collapse short functions to one-liners — entire body fits in 200 chars → one line
8. Column-align one-liner families — related one-liners as vertical table, names/args/bodies column-aligned

**Oracle (REPLACED 2026-07-03, Lon directive — "total mechanical and quick," no eye-check, ever again):** the live build is a unity/jumbo build (`make libscrip_rt` compiles ~130 sources in ONE gcc/g++ invocation straight into the `.so`; no discrete `.o` per file survives, so the original `/tmp/si_objs/*.o` md5 method no longer applies as written — that path doesn't exist in the current build). New method, per touched file: **(1)** get the TRUE original via `git show HEAD:path/to/file.c > /tmp/x.before.c` — **not** a manual `cp` backup, which depends on remembering to make it at the right moment and has already caused one missed fix to go unnoticed for two turns (see Watermark); git is the source of truth, always; **(2)** compile that original standalone — `gcc -c -O0 -w <same -I/-D flags build_scrip.sh/the Makefile use for it> file.c -o before.o` (`g++` for `.cpp`; this project's flags include `-DDYN_ENGINE_LINKED -DIR_DEFINE_NAMES`, easy to forget and easy to get a false compile failure without); **(3)** apply the reformat; **(4)** recompile the same file the same way to `after.o`; **(5)** diff `objdump -d before.o` vs `objdump -d after.o` (instruction stream) AND `objdump -s -j .data -j .rodata -j .bss before.o` vs the same on `after.o` (data sections, whichever are populated) — BOTH must be byte-identical, or the reformat touched semantics and is rejected/fixed. **Known false-positive:** objdump's own output echoes the object's filename in its header (`<name>.o: file format elf64-x86-64`) — normalize it out with a global replace (`sed "s#/tmp/before\.o#OBJ#; s#/tmp/after\.o#OBJ#"` or equivalent), not a line-numbered one (`sed '1s/.../HDR/'` missed it at least twice this session because the real header line isn't always line 1 — sometimes line 2 depending on which objdump mode). **This is the WHOLE gate — corrected 2026-07-03, Lon directive: do NOT also run a full `build_scrip.sh`/`make libscrip_rt` rebuild, and do NOT run any smoke/test/gate script, per touched file OR per batch.** A byte-identical `.o` deterministically implies byte-identical program behavior; re-executing the program adds zero information once the object diff is clean, and this session had been doing it anyway out of habit carried over from other goals where object identity ISN'T the whole story. (A full rebuild remains appropriate at an actual session-end handoff, per RULES.md's own separate handoff protocol — that is a different event with different stakes, not a per-file or per-batch step of this goal.) **After every batch, before reporting it done:** re-run the length/blank census against `git status --short`'s actual modified-file list, not against the list of files you remember touching — this session shipped one false "verified" claim (`arithmetic.c`, two turns ago) that only surfaced because a later, unrelated task happened to trigger a fresh re-check; make the fresh re-check routine, not incidental.
