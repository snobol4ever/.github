## What landed

### RK-OO-A1 — attribute mutation
Was: `$!n = …` (mutate attribute in a method) was a **parse error**; `$obj.x = v` parsed but lowered to a silent no-op (`IR_SUCCEED`) that wrote a *global* named after the field; a void mutator call `$c.bump();` was a **parse error** (the old `class_method` smoke only ever called methods inside `say(...)`).
- **Parser** (`raku.y`): added `VAR_TWIGIL '=' expr ';'` (covers `$.x =` and `$!x =`) and two void method-call statement rules (`VAR_SCALAR '.' IDENT '(' arg_list ')' ';'` and the no-arg form).
- **Lower** (`lower_raku.c`, `lower_rv` TT_ASSIGN): a `TT_FIELD`/`TT_TWIGIL_FIELD` LHS now lowers to `IR_CALL "field_set"` with a `[obj, name, value]` producer chain (mirrors `lower_rcall` visible=1 marshalling; rides the existing native byname path — no template, no new IR kind).
- **Runtime** (`by_name_dispatch.c`): `"field_set"` added to the known-builtin list + handler writing through `data_field_ptr` (the write twin of `dat_field_get`).

### RK-OO-C1/C2/C4 — inheritance
Was: `class Dog is Animal` was a **hard parse error** (no inheritance syntax at all); `DatType` had no parent; `meth_call` did a fixed `Class__method` lookup with no chain walk.
- **C1 parser** (`raku.y`): `KW_CLASS IDENT IDENT IDENT '{' class_body_list '}'` — the middle `IDENT` is checked `== "is"` in the action (see LEXER CONSTRAINT below); parent name stored in the `TT_CLASS_DECL` node's `v.sval` (child name stays in `child[0]`).
- **C2 attribute inheritance** (`driver_data.c`): `void class_inherit(child, parent)` rebuilds the child `DatType.fields` as `[parent fields…, child's own non-dup fields…]` and sets `DatType.parent`. The dup-check makes it **idempotent** (safe to call in m4 where the emitted class-spec already includes inherited fields). `DatType` gained `char parent[64]`. Lowering (`rk_register_classes`) calls it in a second pass after all classes are registered.
- **C4 method inheritance** (`by_name_dispatch.c`): `resolve_method_chain(cls, mname, …)` walks `self → parent → …` via `dat_parent()` and returns the first `<C>__<m>` present in either the proc_table (m2/m3) or the native registry (m4). `meth_call` uses it instead of the old fixed procname build.
- **m4 plumbing** (`scrip.c`): after the `record_register@PLT` loop in `proc_startup`, emit `class_inherit@PLT` calls (child-name + parent-name rodata strings) so the standalone binary repopulates parent links. Construction/field-access need no m4 change because the emitted class-spec already carries flattened fields.

## ⚠️ LEXER CONSTRAINT (important for all future Raku grammar work)
**flex 2.6.4 in this environment CANNOT regenerate `raku.lex.c`** — it fails on the *pristine* (git-HEAD) `raku.l` at line 132 (`<STR_DQ>"\\n"` → "unrecognized rule"). The committed `raku.lex.c` was generated by a different/older flex. Consequence: **do not add new lexer keywords.** Recognize any new keyword at the PARSER level as an ordinary `IDENT` and check the string in the rule action (this is exactly how `is` is handled). `bison -d raku.y -o raku.tab.c` works fine. **Also:** a botched `flex -o raku.lex.c raku.l` will *truncate/delete* the committed lexer — if that happens, `git checkout HEAD -- src/parser/raku/raku.lex.c` to restore it (the lexer is unchanged by this work).

## ⚠️ KNOWN CAVEAT — m4 dup-label on larger OO programs
3+ class programs (e.g. an A←B←C chain with ~6 method-call sites) can trip the **pre-existing `x86_uid` dup-label emitter scaling bug** in m4: assembler error `bbNNNNN_α is already defined`. This is **orthogonal to inheritance logic** — m3 multi-level works perfectly (verified: A←B←C, attrs 1/2/3 + `who`/`ma`/`mb` resolve correctly), and m4 works for the 2-class case (all inheritance smokes pass m4). The dup-label is in the goal-file history as a known emitter limitation; it is the real blocker for larger m4 OO programs and is the thing to fix before scaling OO smokes in m4.
