## Prolog: root cause found, fix needed

### What was fixed

In `parser_prolog.sc`:
1. **Forward-reference ordering**: `push_radix_hex/bin/oct`, `Push_hex/bin/oct_int`, `do_uminus` moved BEFORE `primary` because `primary = (... Push_hex_int('p_radix') ...)` calls them at pattern-build time. Were after `primary`, causing ERROR 022.
2. **Compiland FENCE**: Restructured from `ARBNO($' ' nInc() top_form_safe $' ')` to `ARBNO($' ' FENCE(top_form_safe nInc()) $' ')` — intent: nInc fires only after top_form_safe commits.
3. **Trailing nl strip**: After Src collection loop, added `if (GT(SIZE(Src), 0)) Src = SUBSTR(Src, 1, SIZE(Src) - 1);`.

### The remaining bug

**Root cause (found but not fixed):** In Snocone, `FENCE(top_form_safe nInc())` is a **function call** — `FENCE` is called immediately when `Compiland = ...` is evaluated, with `top_form_safe nInc()` as its argument. This means `nInc()` fires at **pattern-build time** (once, during init), not at match time. So the FENCE approach doesn't solve the nInc corruption problem.

**Correct fix (implement in next session):** Move `nInc()` **inside** `top_form_safe` — i.e., inside `top_form_counted`:

```snocone
/* In parser_prolog.sc, replace the Compiland block: */
top_form_counted = nInc() top_form_safe;
Compiland = nPush()
            POS(0) ARBNO( $' ' *top_form_counted $' ' ) RPOS(0)
            reduce(E_Parse, 'nTop()')
            nPop();
```

The `*top_form_counted` defers the entire pattern (including `nInc()`) until match time. The `*` (defer) operator is the key. When `top_form_counted` fails (trailing newline), the deferred `nInc()` never fires because pattern matching backtracks before executing the side-effect.

**BUT** there's a subtlety: `nInc()` returns pattern `epsilon . *IncCounter()`. The `*IncCounter()` is already deferred. So `nInc() top_form_safe` = concat of `epsilon . *IncCounter()` and `top_form_safe`. If this concat fails after `IncCounter()` has incremented, we still have the corruption. The real fix is **deferred evaluation of the whole unit**.

**Simplest correct approach:** restructure `top_form_safe` to include `nInc()` at the commitment point of each sub-form:

```snocone
/* Add nInc() at the START of directive, clause, dcg_rule: */
directive = ( nInc() $':-' Reset_var_scope body $'.' Build_directive );
clause    = ( nInc() head (($':-') body Mark_body | epsilon) $'.' Build_clause );
dcg_rule  = ( nInc() head $'-->' Mark_body dcg_body $'.' Build_dcg );
skip_to_dot = ( nInc() BREAKX('.') $'.' Push_skip );

/* And simplify Compiland back to: */
Compiland = nPush()
            POS(0) ARBNO( $' ' top_form_safe $' ' ) RPOS(0)
            reduce(E_Parse, 'nTop()')
            nPop();
```

This mirrors exactly how snocone works: each Command sub-rule (`if_cmd`, `stmt_cmd` etc.) begins with `nInc()`, so if the sub-rule's first token doesn't match, `nInc()` never fires.

The `nInc()` at position 1 of each sub-rule fires when SPITBOL can't backtrack past it (because the rule committed). But actually, the `$':-'` and `head` at positions 1-2 are the true commitment points. Moving `nInc()` to position 1 still risks firing before commitment.

**Truly safe fix:** use `nInc()` AFTER a commitment:
```snocone
directive = ( $':-' nInc() Reset_var_scope body $'.' Build_directive );
```

When `$':-'` matches and consumes `:-`, SPITBOL is committed to this being a directive. `nInc()` fires. If `$':-'` doesn't match, `nInc()` never fires. This is the exact pattern used by snocone's grammar.
