### IJ-LCONCAT — `|||` list/string concat operator (`e8edc709`)

TT_LCONCAT had no lowering case in lower_icn.c so any `E1 ||| E2`
fell through to ICN_BB_EVAL → `[NO-AST] SM_BB_EVAL` stub at runtime.
The runtime helper bb_eval_value in icn_value.c already had correct
list-vs-string dispatch — just needed to factor it out for IR.

Four coherent edits:

1. **`src/include/IR.h`** adds IR_ICN_LCONCAT after IR_INITIAL.
2. **`src/runtime/interp/icn_value.{c,h}`** factors reusable DESCR_t-
   input helpers `icn_str_concat_d(a,b)` and `icn_lconcat_d(a,b)` out
   of the existing tree_t*-taking bb_str_concat / TT_LCONCAT block.
   List-list builds fresh icnlist; otherwise falls back to string
   concat with numeric coercion. Both helpers exported.
3. **`src/lower/ir_exec.c`** includes icn_value.h and adds IR_ICN_LCONCAT
   executor right after IR_INITIAL. Evaluates both children, propagates
   either-side FAIL via ω, then dispatches via icn_lconcat_d. Single-
   shot — IR_ICN_LCONCAT is NOT a generator kind even with generative
   operands (Icon's spec does not cross-product over `|||`).
4. **`src/lower/lower_icn.c`** adds TT_LCONCAT lowering case right
   after the TT_CAT/binop block.

Plus IR_ICN_LCONCAT entry in scrip_ir.c kind_names.

**Result:** ir-run 191 → 192 (+1). rung15_real_swap_lconcat PASS.
smoke_icon 5/5.

### IJ-FIND-GEN — `find()` as resumable generator (`158144eb`)

Icon's `find()` is spec'd as a generator yielding every match position,
but the IR path routed all calls through `icn_try_call_builtin_by_name`
which returns the first match only. Five coherent edits:

1. **`src/include/IR.h`** adds IR_ICN_FIND_GEN after IR_ICN_LCONCAT.
2. **`src/lower/ir_exec.c`** adds IR_ICN_FIND_GEN executor with α/β:
   - α (state==0): evaluate needle, hay, optional start (1-based,
     default 1), optional stop (1-based exclusive, default len+1).
     Normalize 0/negative bounds per Icon spec (0=end, negative=
     from-end). Cache needle/hay strings + stop in a GC_malloc'd
     struct stashed in opaque so β doesn't re-evaluate children
     and re-fire any side effects. counter holds 0-based search-
     start position.
   - β (state==1): search from counter via strstr; if hit, return
     1-based position and update counter to advance one past this
     match's start. If miss or past stop, FAIL via ω.
   Empty-needle case (find("",h)): yield each 1..stop position once.
3. Same file adds IR_ICN_FIND_GEN to the three generator-kind tables:
   `ir_is_single_shot` exclusion list, `IR_IS_GEN_KIND` macro (BINOP_GEN
   cross-product re-pump), `ALT_IS_GEN` macro (IR_ALT beta-resume).
4. **`src/lower/lower_icn.c`** adds find() special-case in TT_FNC
   right after key(). Only 2/3/4-arg forms (find(n,h), find(n,h,start),
   find(n,h,start,stop)) lower to IR_ICN_FIND_GEN. The 1-arg form
   (find(n) — search in &subject) falls through to IR_CALL because
   scan-context resolution lives downstream.

Plus IR_ICN_FIND_GEN entry in kind_names.

**Result:** ir-run 192 → 193 (+1). rung08_strbuiltins_find_gen PASS
(`every write(find("a","banana"))` yields 2, 4, 6). smoke_icon 5/5.

**Carry-forward:** start/stop bounds normalization for negative and
zero inputs is written but untested against jcon corner cases — may
surface issues in rung36 jcon programs that use 4-arg find().

### IJ-SEQ-GEN — `seq()` as infinite arith-progression generator (`f81302ca`)

Icon's `seq()` is spec'd as an infinite generator yielding start, start+
step, start+2*step, ... typically paired with `\ N` (IR_LIMIT) to bound
it. Like find, seq was in `ir_is_single_shot`'s known-generators
exception (line 65) but the actual IR path delivered only the first
value via `icn_try_call_builtin_by_name` → interp_eval's single-shot
`return IS_INT_fn(start) ? start : INTVAL(1)`.

Five coherent edits (paralleling IJ-FIND-GEN):

1. **`src/include/IR.h`** adds IR_ICN_SEQ_GEN after IR_ICN_FIND_GEN.
2. **`src/lower/ir_exec.c`** adds IR_ICN_SEQ_GEN executor right after
   IR_ICN_FIND_GEN:
   - α (state==0): read optional start (c[0], default 1) and step
     (c[1], default 1). Int-coerce from real if needed. Zero step
     silently bumped to 1 (avoid no-progress). Cache step in nd->ival;
     cache start in nd->counter. Yield start via γ.
   - β (state==1): counter += ival; yield via γ.
   No upper bound — relies on consumer (IR_LIMIT via `\N`, every with
   break) to terminate. Without limiter, infinite loop intended.
3. Same file: add IR_ICN_SEQ_GEN to ir_is_single_shot exclusion,
   IR_IS_GEN_KIND macro, ALT_IS_GEN macro.
4. **`src/lower/lower_icn.c`** adds seq() lowering case after find()
   in TT_FNC. Handles 0/1/2-arg forms; falls back to IR_CALL for any
   other arity.

Plus IR_ICN_SEQ_GEN entry in kind_names.

**Result:** ir-run 193 → 194 (+1). rung30_builtins_misc_seq PASS
(`every write(seq(1) \ 3)` yields 1, 2, 3). All gates hold.

## Open Notes / Carry-Forward

1. **IR_ICN_FIND_GEN start/stop normalization untested.** The 0/negative
   bound logic (0=end, negative=from-end, with separate normalization
   for start vs stop) is written but only the 2-arg form was exercised
   in the gate. If a rung36 jcon program uses `find(n,h,start,stop)`
   with negative indices, that's the place to look first if results
   are wrong.

2. **IR_ICN_SEQ_GEN ignores zero step.** Icon's spec says zero step
   should FAIL the seq call; my implementation silently bumps it to
   1 to avoid infinite-no-progress hangs. If anything strictly checks
   this behavior, the executor's `if (step == 0) step = 1;` is the
   trivial place to change to `if (step == 0) { nd->value = FAILDESCR;
   return nd->ω; }`.

3. **The DESCR_t-helper refactor in icn_value.c** factored the list-vs-
   string dispatch into reusable `icn_lconcat_d`. The original tree_t*-
   taking TT_LCONCAT case in bb_eval_value still has the same logic
   inlined — it was NOT refactored to call icn_lconcat_d because
   that would change the behavior of mode-1 AST-walk semantics, which
   could ripple. Two paths now exist with identical semantics. If
   anyone changes one, mirror the other.

4. **IR_ICN_FIND_GEN opaque uses GC_malloc.** The struct's pointer-
   typed members (needle, hay) point to strings owned elsewhere
   (interned in the AST or in a STRVAL DESCR_t buffer). As long as
   those source strings live for the duration of the every-loop
   (they do — they're GC-anchored via the IR graph's value field),
   no use-after-free. But a future migration that frees AST early
   would need to deep-copy these into the opaque struct.

5. **`every (s := "" | "a") do write(s)`** — listed in 17d carry-forward
   for re-verification. Still not exercised this session; likely fine
   given the loop body-state-reset that landed in 17d but worth
   confirming when tests are around.
