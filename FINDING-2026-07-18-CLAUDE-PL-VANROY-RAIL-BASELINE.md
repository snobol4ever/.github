## 2. THREE METHOD CORRECTIONS (all measured, none guessed)
1. **The FINDING-2026-07-10 recursion-loop wrapper shape is UNUSABLE on gprolog.** gprolog has NO heap GC — only backtracking reclaims the global stack — so a deterministic recursion loop accrues every iteration's structures until the global stack dies MID-LOOP, silently: fib ×1000 emitted 42 lines then stopped (wall 599 ms, no visible error through the harness filter); GLOBALSZ=262144 raised it only to 191. The rail ships the classic FAILURE-DRIVEN shape instead: `l__(N) :- between(1,N,_), bench__main, fail. l__(_).` — per-iteration reclamation by backtracking on every engine.
2. **`once/1` as the per-iteration fence EXPOSED A SCRIP CORRECTNESS BUG.** Reproducer: queens_8 wrapped `between(1,50,_), once(bench__main), fail` → GNU prints 50 lines; SCRIP m3 printed 1111 (≈ full 92-solution enumeration per outer step) — SCRIP's once does NOT hold its cut-commit when redone from OUTSIDE. Not hunted this session (MONITOR-FIRST applies when it is); banked here with the reproducer. The rail therefore runs WITHOUT once — full enumeration per iteration, identical work on every engine, still a fair comparison.
3. **N is AUTO-RANGED PER ENGINE** (×4 until compute ≥300 ms, cap 65536; timeout 240 s ⇒ DNF), not one shared gprolog-calibrated N: per-iteration ms is N-independent at steady state, and one N across a 100–4000× spread either starves gprolog's precision (wall ≈ floor noise) or blows SCRIP's wall clock. Consequence stated honestly: SCRIP cells stop at N=1..64, i.e. SCRIP is measured PRE-accrual; its true steady state is WORSE (see §4b).
